# microsoft365_graph_beta_identity_and_access_network_filtering_profile_policy_link (Resource)

Manages a Microsoft Entra Global Secure Access filtering profile policy link using the Microsoft Graph beta `/networkAccess/filteringProfiles/{filteringProfileId}/policies` endpoint.

This resource manages a Microsoft Entra Global Secure Access **Security profile** policy link. A policy link attaches an existing network access policy to an existing filtering profile.

## Microsoft Documentation

- [policyLink resource type](https://learn.microsoft.com/en-us/graph/api/resources/networkaccess-policylink?view=graph-rest-beta)
- [filteringPolicyLink resource type](https://learn.microsoft.com/en-us/graph/api/resources/networkaccess-filteringpolicylink?view=graph-rest-beta)
- [cloudFirewallPolicyLink resource type](https://learn.microsoft.com/en-us/graph/api/resources/networkaccess-cloudfirewallpolicylink?view=graph-rest-beta)
- [threatIntelligencePolicyLink resource type](https://learn.microsoft.com/en-us/graph/api/resources/networkaccess-threatintelligencepolicylink?view=graph-rest-beta)

## Microsoft Graph API

This resource uses the beta `/networkAccess/filteringProfiles/{filteringProfileId}/policies` endpoint.

Microsoft Learn documents `policyLink` as an abstract type with `filteringPolicyLink`, `cloudFirewallPolicyLink`, `threatIntelligencePolicyLink`, and `tlsInspectionPolicyLink` derived types. The Entra admin center was also observed linking web filtering policies with `#microsoft.graph.networkaccess.webFilteringPolicyLink`, which is not currently generated in the Microsoft Graph beta Go SDK. The Entra admin center JavaScript bundle also defines portal-first `promptPolicyLink`, `filePolicyLink`, and `securityProviderPolicyLink` types. The provider sends explicit `@odata.type` values so both documented link types and portal-first link types can be managed.

Observed Entra admin center traffic for creating a legacy filtering policy link uses:

```http
POST /beta/networkaccess/filteringProfiles/{filteringProfileId}/policies
```

with a `#microsoft.graph.networkaccess.filteringPolicyLink` body containing `priority`, `state`, `loggingState`, and a nested `#microsoft.graph.networkaccess.filteringPolicy` reference.

Observed Entra admin center traffic for creating a web filtering policy link uses the same collection endpoint with a `#microsoft.graph.networkaccess.webFilteringPolicyLink` body containing `state` and a nested `#microsoft.graph.networkaccess.webFilteringPolicy` reference.

Observed Entra admin center traffic for creating a content policy link uses the same collection endpoint with a `#microsoft.graph.networkaccess.filePolicyLink` body containing `state` and a nested `#microsoft.graph.networkaccess.filePolicy` reference. This Graph type naming does not match the UI label; Terraform exposes it as `content`.

Observed Entra admin center traffic for selecting Netskope DLP policies queries `/networkAccess/securityProviderPolicies` with a Netskope schema filter. Terraform exposes this as `netskope_dlp` and maps it to `#microsoft.graph.networkaccess.securityProviderPolicyLink` internally.

## Behavior Notes

- This resource links existing policies to an existing Security profile. It does not create or update the filtering profile or the policy itself.
- `policy_link_id` is generated by Microsoft Graph and differs from `policy_id`.
- Import uses `{filtering_profile_id}/{policy_link_id}`.
- `priority` and `logging_state` are used only for `filtering` links and must be omitted for other `policy_type` values. Other link types use the common `policyLink` fields.
- Link timestamp fields are intentionally not exposed. Microsoft Learn documents `createdDateTime` and `lastModifiedDateTime` on `filteringPolicyLink`, but not on the common `policyLink` base type or the other documented link types; portal-observed web filtering links return those fields as `null`.
- Microsoft Graph `@odata.type` values are intentionally hidden from the Terraform schema. Select a supported `policy_type`; the provider maps it to the Graph beta link and policy types internally. When a new portal-first policy type appears in Entra XHR, this resource should add a new `policy_type` value and internal mapping rather than exposing raw OData type strings to users.

## Example Usage

```terraform
resource "microsoft365_graph_beta_identity_and_access_network_filtering_profile" "example" {
  name        = "Example Security Profile"
  description = "Global Secure Access security profile managed by Terraform"
  priority    = 100
  state       = "enabled"
}

resource "microsoft365_graph_beta_identity_and_access_network_web_filtering_policy" "example" {
  name           = "Example Web Filtering Policy"
  description    = "Global Secure Access web filtering policy managed by Terraform"
  default_action = "allow"
}

resource "microsoft365_graph_beta_identity_and_access_network_filtering_profile_policy_link" "web_filtering" {
  filtering_profile_id = microsoft365_graph_beta_identity_and_access_network_filtering_profile.example.id
  policy_id            = microsoft365_graph_beta_identity_and_access_network_web_filtering_policy.example.id
  policy_type          = "web_filtering"
  state                = "enabled"
}
```

## Schema

### Required

- `filtering_profile_id` (String) The ID of the filtering profile, shown as a security profile in the Microsoft Entra admin center.
- `policy_id` (String) The ID of the existing policy to link to the filtering profile.
- `policy_type` (String) The policy type to link. Known values are `filtering`, `web_filtering`, `prompt`, `content`, `netskope_dlp`, `cloud_firewall`, `threat_intelligence`, and `tls_inspection`.

### Optional

- `logging_state` (String) The logging state for a `filtering` link. The Entra admin center sends this for legacy filtering policy links and does not send it for `web_filtering` links. Omit this for non-`filtering` links.

- `state` (String) The state of the policy link. Possible values are `enabled` and `disabled`.
